## Context

D3/CER-055 (agreements `HARNESS007-main.md` DP5). Waypoint outcomes render uniformly **FAIL**
across 224 / 460 attempts on radar/forqsite — implausible. The fault is either the source of
truth (`record_attempt.py` not writing the `outcome` column for non-reviewer rows — builder/leaf
rows may carry NULL `outcome`, which the SPA renders as FAIL) or the render
(`effortDb.ts queryWaypoints` / `context.ts` mapping NULL → FAIL). This story fixes both ends
and re-verifies the `pairmode_effort.py models` PASS-rate report.

## Requires

- HARNESS006 complete: leaf-worker rows are recorded per `agent_role` (the rows whose `outcome`
  the SPA reads).

## Ensures

- **Source of truth:** `record_attempt.py` records a correct `outcome` for the rows that should
  carry one (the leaf-worker outcome — e.g. the reviewer PASS/FAIL, and any builder/leaf row
  that has a defined outcome); rows with no defined outcome carry an explicit NULL, never a
  defaulted FAIL.
- **Render:** `effortDb.ts queryWaypoints` / `context.ts` map a NULL/absent `outcome` to a
  neutral state (e.g. "n/a"/"pending"), **never FAIL**; only an actual `FAIL` value renders FAIL.
- The `pairmode_effort.py models` PASS-rate report is re-verified uncorrupted (a deterministic
  test over a synthetic effort.db with mixed PASS/FAIL/NULL rows asserts the PASS-rate is
  computed from real outcomes, not NULL-as-FAIL).
- `tests/pairmode/test_waypoint_outcome.py` asserts deterministically (synthetic effort.db):
  builder/leaf rows without a defined outcome are NULL not FAIL; the waypoint query/render maps
  NULL → neutral; a real FAIL renders FAIL; the PASS-rate report is correct. No live fleet
  dependence.
- `PATH=$HOME/.local/bin:$PATH uv run pytest tests/pairmode/ -x -q` passes; the API
  builds/typechecks.

## Instructions

- Determine which is wrong (recording vs render) with a synthetic effort.db and assert both
  ends; fix both if both are wrong. The binding rule: **NULL outcome ≠ FAIL** everywhere
  (writer default and reader map).
- Do not change the effort.db schema (no new column) — this is a value-correctness fix on the
  existing `outcome` column.
- Re-run/verify the `pairmode_effort.py models` report path in the test.
