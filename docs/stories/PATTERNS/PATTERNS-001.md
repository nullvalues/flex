## Pattern identity

**Name:** Builder/Reviewer Sub-Agent Loop
**Category:** Agentic Architecture
**One-line intent:** A closed spec→build→review→re-spec loop where builder and reviewer
are separate sub-agents reading from the same phase doc, and the reviewer's only
authority is the spec — not opinion — ensuring correctness is validated against recorded
intent on every commit.
**Also Known As:** Spec-Gated Review Loop, Closed-Loop QA Sub-Agent, Reviewer-As-Spec-Enforcer

**Distinguish from** `quality-gate-checkpoint` (catalog): that pattern is about
a single agent self-reviewing email drafts before human approval; this pattern is about
two *separate* sub-agents operating a closed development QA loop where the spec is the
source of truth and the loop exits only when the reviewer signs off the BUILD GATE.

## What broke (cite these real incidents)

- **CER-027** (2026-05-29, Phase 47): The context budget check was LLM-attention-based
  only — the rule lived in a doc the agent read at session start and was expected to honor
  each iteration. Confirmed failure mode: the LLM moved quickly through stories, skipped
  the check, context bloated past threshold, and the prompt that should have appeared
  at story boundaries never fired. "The architect of the rule cannot reliably follow it."
  Resolution: mechanical enforcement via pre_tool_use.py hook. This failure mode is the
  canonical example of why spec-authority must be mechanically enforced, not attention-based.
- **CER-028** (Phase 47): BUILD GATE variable inconsistency surfaced — reviewers were
  checking different conditions than the spec required because there was no single spec
  artifact the reviewer read from. Read `docs/cer/backlog.md` for the full entry.
- Agents drifting to taste-based code review: without the spec-only authority constraint,
  reviewers add scope, impose personal preferences, or silently accept half-finished work
  (pattern observed across early flex phase reviews before the methodology was formalized).

Key sections with content guidance:

**Pattern in 60 Seconds:** Problem = agents producing code without spec accountability.
Insight = separate sub-agents + spec-as-sole-authority eliminates drift. Key structure =
three-row table (Builder / Reviewer / Orchestrator) with roles and constraints. What broke
= CER-027 context overrun from attention-based enforcement.

**Classification table:** Category: Agentic Architecture. Difficulty: Advanced.

**Motivation:** Describe a concrete scenario where an agent reviews code against its own
taste rather than a spec, causes scope creep, and the human has no way to audit the
decision. Then contrast with the flex pairmode build loop.

**Applicability — Use when:**
- Code is being generated by an agent from a phase spec
- The spec is the canonical source of acceptance criteria
- You want to prevent taste-based or opinion-based gate failures
- The project spans multiple sessions (spec continuity is needed)

**Applicability — Do NOT use when:**
- There is no pre-existing spec (the pattern requires one)
- Tasks are exploratory/research (no acceptance criteria to enforce)
- The build loop overhead exceeds the value of spec enforcement (tiny one-off scripts)

**Structure:** Mermaid sequenceDiagram showing: Orchestrator reads phase spec → spawns
Builder (with story) → Builder implements → Orchestrator commits methodology files →
Orchestrator spawns Reviewer (with story + spec) → Reviewer diffs + runs checklist +
runs tests → Reviewer commits (PASS) or reverts (FAIL) → Orchestrator handles result.

**Participants table:** Orchestrator / Builder sub-agent / Reviewer sub-agent /
Phase spec / Story file / BUILD GATE / Test suite.

**How It Works:** Number the steps: (1) Orchestrator reads next planned story, (2)
Spawns builder with story text, (3) Builder implements and stops (no commit), (4)
Orchestrator commits methodology files pre-reviewer, (5) Spawns reviewer with story
spec, (6) Reviewer runs checklist against spec only, (7) Reviewer commits (PASS) or
reverts (FAIL), (8) On FAIL: auto-retry with escalated model, then loop-breaker.

**Consequences — Benefits:** Scope creep eliminated (reviewer is spec-bound). Opinion
drift eliminated. Every commit is provably tied to a story. Model escalation on retry
recovers most failures without human intervention.

**Consequences — Liabilities:** Requires a pre-existing spec. Loop overhead per story
(two sub-agent spawns). Reviewer can only enforce what is in the spec — if the spec is
wrong, the reviewer enforces the wrong thing.

**Consequences — What Broke in Practice:** Cite CER-027 and CER-028 from above.

**Security Implications:** Attack surface: builder has write access to declared story
scope; reviewer has read-only tools + Bash. Failure modes: builder escapes declared
scope (mitigated by pre-authorized write permissions cleared after each story). Data
sensitivity: story files and phase docs contain project decision history — treat as
internal confidential.

**Known Uses:** flex project — used on every code story across Phases 10–47,
7 downstream projects (forqsite, radar, asp, aab, cora, and others).

**Related Patterns:**
- `checkpoint-gated-autonomy`: the loop's BUILD GATE IS the autonomy checkpoint
- `hub-and-spoke-orchestration`: orchestrator is the hub; builder/reviewer are spokes
- `quality-gate-checkpoint`: superficially similar but distinct (see "Distinguish from" above)
- `runbook-driven-agent-cadence`: both separate spec from execution; this pattern's
