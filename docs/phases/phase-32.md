# anchor — Phase 32: Story-as-contract and story_context CLI

← [Phase 31: Discoverability and status panel](phase-31.md)

## Goal

Two related improvements that tighten the build loop:

1. **Story-as-contract** (OpenProse-inspired): Replace the narrative `## Acceptance
   criterion` section with two structured sections — `## Requires` (preconditions the
   builder depends on) and `## Ensures` (binary assertions the reviewer verifies
   independently). Each `## Ensures` item is checkable without interpretation, turning
   the reviewer's acceptance check from a judgment call into a mechanical verification.
   Backward-compatible: existing stories with `## Acceptance criterion` continue to work.

2. **story_context.py CLI**: The session-start hint and bootstrap next-steps both
   reference `story_context.py --set RAIL-NNN`, but `story_context.py` has no CLI
   entry point. This phase adds one.

Plus the one genuinely unimplemented lesson from the lessons review: **L008** — a
pre-build spec review step in the orchestrator, so architectural errors in a phase spec
are caught before builder time is spent on them.

These four stories have one dependency: INFRA-075 (reviewer contract check) must build
after INFRA-074 (contract sections format), since the reviewer template needs to know
the exact section structure to verify. INFRA-076 and INFRA-077 are independent.

### Story INFRA-075 — Reviewer contract verification step

**Rail:** INFRA | **story_class:** methodology

## Requires
- INFRA-074 complete: `## Requires` / `## Ensures` format is defined and documented.

## Ensures
- `skills/pairmode/templates/agents/reviewer.md.j2` has a "Contract check" section
  before the review checklist.
- The contract check reads `## Ensures` items from the story spec (if present) and
  verifies each as an independent binary assertion, reporting `PASS` or `FAIL` per
  item with the item text.
- When no `## Ensures` section is present (legacy story), the contract check is
  skipped and the reviewer falls back to the narrative `## Acceptance criterion`
  judgment as before.
- `skills/pairmode/templates/agents/reviewer.md.j2` template change is also applied
  to anchor's own `.claude/agents/reviewer.md` (via `pairmode sync-agents` after
  commit, or directly).

**Instructions:**

1. Read `skills/pairmode/templates/agents/reviewer.md.j2` in full. Find `## Before
   reviewing` (the pre-review steps) and `## Review checklist`.

2. Insert a new `## Contract check` section between `## Before reviewing` and
   `## Review checklist`:

   ```markdown
   ## Contract check

   Read the story spec's `## Ensures` section (if present).

   If `## Ensures` is present:
     For each item listed under `## Ensures`:
     - Verify the assertion independently (read the file, run the command, check the
       output). Do not read the item and assume it is satisfied.
     - Report: `ENSURES [n]: PASS — <item text>` or `ENSURES [n]: FAIL — <item text>
       — <what you found instead>`.
     - A single FAIL here is a contract violation. Set overall verdict to FAIL.
   
   If no `## Ensures` section (legacy story with `## Acceptance criterion`):
     Skip this section. The acceptance criterion check is handled narratively in the
     checklist.
   ```

3. Update the commit message format line in the template to reference `## Ensures`
   alongside `## Acceptance criterion` so both formats are acknowledged.

4. Run `pairmode sync-agents --project-dir . --yes` to propagate the updated template
   to anchor's own `.claude/agents/reviewer.md`.

**Primary files:** `skills/pairmode/templates/agents/reviewer.md.j2`
**Touches:** `.claude/agents/reviewer.md`, `docs/architecture.md`

**Tests:** Methodology story — no test file expected. Verify by reading the updated
template and confirming the contract check section is present, correctly positioned,
and handles both the `## Ensures` and legacy `## Acceptance criterion` cases.

### Story INFRA-076 — `story_context.py` CLI

**Rail:** INFRA | **story_class:** code

## Requires
- `story_context.py` library functions (`set_current_story`, `get_current_story`,
  `clear_current_story`) exist and work correctly.
- `bootstrap.py` `_print_next_steps` currently references a broken
  `story_context.py --set` invocation (corrected to `story_new.py` in Phase 31,
  but a proper CLI is still absent).

## Ensures
- `uv run python skills/pairmode/scripts/story_context.py --set INFRA-074` writes
  `INFRA-074` as `current_story.id` in `.companion/state.json` and exits 0.
- `uv run python skills/pairmode/scripts/story_context.py --get` prints the current
  story ID to stdout (or "No story set." if absent) and exits 0.
- `uv run python skills/pairmode/scripts/story_context.py --clear` removes
  `current_story` from `.companion/state.json` and exits 0.
- `bootstrap.py` `_print_next_steps` step 1 references the correct
  `story_context.py --set RAIL-001` invocation.
- The session-start hook (or `pairmode_status.py`) hint "Set one with:
  `story_context.py --set RAIL-NNN`" now points to a working command.

**Instructions:**

1. In `skills/pairmode/scripts/story_context.py`, add a Click CLI entry point at the
   bottom of the file:
   - `--set STORY_ID` (string): resolves the story file from
     `docs/stories/<RAIL>/<RAIL>-NNN.md` (where RAIL is the uppercase prefix of
     STORY_ID). Reads frontmatter to get title and status. Calls
     `set_current_story(companion_dir, story_id, {"id": story_id, "title": title,
     "status": status})`. Prints `Story set: STORY_ID`.
   - `--get` (flag): calls `get_current_story(companion_dir)`. Prints the `id` field
     if set, or "No story set." if absent.
   - `--clear` (flag): calls `clear_current_story(companion_dir)`. Prints
     "Story cleared.".
   - `--project-dir PATH` (optional, default "."): used to locate `.companion/`.
   - Mutually exclusive: exactly one of `--set`, `--get`, `--clear` must be provided.

2. In `skills/pairmode/scripts/bootstrap.py`, update `_print_next_steps` step 1 to
   reference the correct CLI:
   ```
   1. Set your current story:
        uv run python skills/pairmode/scripts/story_context.py --set RAIL-001 \
          --project-dir {project_dir}
   ```

3. Verify the session-start hint in `.companion/state.json` handling or
   `pairmode_status.py` already says `story_context.py --set RAIL-NNN` and confirm
   it now points to a working command (no code change needed if wording already
   matches; update if it references a different command).

**Primary files:** `skills/pairmode/scripts/story_context.py`
**Touches:** `skills/pairmode/scripts/bootstrap.py`,
`tests/pairmode/test_story_context.py`

**Tests:** `tests/pairmode/test_story_context.py` — add CLI tests using Click's
`CliRunner`:
- `--set INFRA-001` with a fixture story file writes `current_story` to state.json.
- `--get` returns the story ID when set; "No story set." when absent.

### Story INFRA-077 — Pre-build spec review step (L008)

**Rail:** INFRA | **story_class:** methodology

## Requires
- `CLAUDE.build.md` and `CLAUDE.build.md.j2` exist and pass template tests.

## Ensures
- `CLAUDE.build.md` has a `### 0. Spec review (optional)` step inside `## Before
  the first build loop`, placed after step 7 (story identification) and before
  `## Model evaluation`.
- The step instructs: spawn a `general-purpose` subagent to cold-eyes review the
  full phase spec against the actual codebase; incorporate any CRITICAL or HIGH
  findings before building; skip for single-story hotfix phases.
- The same step appears in `skills/pairmode/templates/CLAUDE.build.md.j2`.
- `docs/architecture.md` notes the spec review step under the build loop description.
- Lesson L008 is marked `applied` in `lessons/lessons.json`.

**Instructions:**

1. In `CLAUDE.build.md`, after step 7 in `## Before the first build loop` (the step
   that resolves the story ID from the phase manifest) and before `## Model
   evaluation`, add:

   ```markdown
   ### 0. Spec review (optional but recommended for phases with 3+ stories)

   Before spending builder time, cold-eyes review the phase spec against the codebase.
   Spawn a `general-purpose` subagent with:
   - The full phase spec text (from the phase file)
   - Instruction: "Review this phase spec against the actual codebase at [project root].
     Find: mismatched function signatures, missing imports, wrong call-site arguments,
     references to non-existent files or symbols, type mismatches. Report CRITICAL and
     HIGH findings only."

   Incorporate any CRITICAL or HIGH findings by updating the phase spec before building
   the first story. LOW/MEDIUM findings: note them but proceed.

   Skip this step for: single-story hotfix phases, documentation-only phases, or when
   the phase spec was already reviewed in the previous session.
   ```

2. Apply the identical change to `skills/pairmode/templates/CLAUDE.build.md.j2`.

3. In `docs/architecture.md`, add a one-line note about the optional spec review step

   `lessons/LESSONS.md` by running:

   or the equivalent lessons regeneration command.
